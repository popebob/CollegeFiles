PROBLEM STATEMENT

 

You are a programmer for the Condo Management Company. The company is located in a ski resort community and provides a rental service for condo owners who want to rent their units. The company rents the condos by the week to ski vacationers.

 

Currently the company has a manual system that is paper-based, time-consuming and prone to errors. The company expects to grow significantly in the near future and they need to computerize their record processing system. You have been assigned the task of building a database, using Microsoft Access, to replace the paper system.

DATABASE STRUCTURE

The Condo Management company keeps condo owner data such as name, address and contact information in their files. A condo owner may have more than one condo available. They also keep detailed descriptions of each of the condos that are available for rent including such things as number of bedrooms, number of baths, and which amenities are included as well as what restrictions may apply. Information about weekly rental rates is also recorded. The following information is currently kept by the company about their rental properties:

  • owner's last name

  • owner's first name

  • owner's address

  • owner's phone number

  • number of bedrooms

  • number of bathrooms

  • list of amenities (linens, Jacuzzi, fireplace)

  • list of restrictions (pets, smoking, maximum number of people)

  • weekly rental rate

The first step in designing a database is to analyze the data that will be stored in the database. Tables are used to store data in a database. Each table will contain only one kind of data. Tables consist of records which contain fields, one field for each piece of data to be stored. Each table must have one field that contains unique data (the data in the field is never repeated in the table); this field is called the table's primary key field.

 

Careful study of the list above reveals the following facts:

  • The first 4 items are owner data.

  • The remaining items are condo unit data.

  • In order to avoid repeating all of the owner data for clients offering multiple units, it is most efficient to store owner data separate from condo data.

  • Our database will need two tables. One will be used to store owner data; the other will be used to store condo data.

  • Storing owner data in a separate table from condo data requires that one of the tables include some kind of cross-reference identification in order to associate the condo data with the correct owner data.

  • There is no identifiable primary key field among the owner data; none is unique. Unique data must be "invented" for each owner. This is typically done by assigning a unique code to each owner.

  • Also, none of the condo data is unique. Unique data must therefore be "invented" for each condo unit. As with owners, this can be done by assigning a unique code to each condo.

  • Because an owner may own more than one rental unit, there is a "one-to-many" relationship between owners and condos.

  • Because of this one-to-many relationship, the most efficient way to associate ("join") the data in the two tables is to repeat the unique owner code in each of his condo table records. This repeated field is called a foreign key since it is the primary key of another table. It must contain contain exactly the same data as the primary key.

The second step in database design is to look very carefully, again, at the list of data that needs to be stored in the database. Further analysis most often results in a much more detailed list of fields that must be included in the database tables. For example, because of postal regulations, one field containing all address data is not suitable. Address information must be broken down into several entities. Similarly, the original lists of amenities and restrictions will need to be broken into multiple fields in the final design of the condo database table.

 

The third step in database design is to assign data attributes to each of the identified fields. We need to determine 4 attributes - field name, data type, field size, and primary key status. Some of the questions that need to be asked are:

  • Will the field contain text? If so, how many characters will be stored?

  • Will the data be used in mathematical calculations? If so, how large a number must be stored; how many decimal places are required?

Microsoft Access offers a number of data types. As we gain experience with different types of data, you will quickly be able to judge which is best to use. For the initial table structures, we will use the attributes included in the following two tables:
